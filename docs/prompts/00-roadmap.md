# Roadmap de prompts

Cada uno de los siguientes ítems será un bloque de prompt independiente que el usuario hará copy-paste a Copilot cuando llegue su turno. Este archivo es el índice maestro.

| # | Archivo | Objetivo |
|---|---------|----------|
| 01 | `01-init-laravel.md` | Inicializa Laravel 12 + Pest + Tailwind + Vite. `git init` local. Primer commit incluye toda la documentación pre-existente (ARCHITECTURE.md, DESIGN.md, ADRs, README.md, copilot-instructions.md). Push a `github.com/blancosampedro-ship-it/pivscrtm.git`. **Añade `Route::get('/up')` health check** y configura `tailwind.config.js` con tokens de DESIGN.md (cobalto + Instrument Serif + General Sans desde Bunny Fonts). |
| 01b | `01b-github-actions.md` | GitHub Actions CI: workflow que en cada push corre `composer install`, `./vendor/bin/pint --test`, `./vendor/bin/pest`. Falla si Pint encuentra issues o si algún test falla. ~30 min de setup, blinda regresiones desde el día 1. |
| 02 | `02-env-and-db.md` | Configura `.env` local apuntando a SiteGround Remote MySQL. Whitelisting de IP en Site Tools. Probar conexión y listar las 14 tablas existentes. **Cuatro tareas críticas adicionales:** (1) **Borrar el dump SQL público** `https://winfin.es/serv19h17935_winfin_2025-04-25_07-53-24.sql` AHORA, no esperar a Fase 7 (incidente RGPD activo cada día que sigue online). (2) Verificar con SQL de ADR-0005 §4 si hay emails duplicados entre `u1`/`tecnico`/`operador` (dato necesario para el bloque 6). (3) **Verificar el cron real** de SiteGround: SSH `crontab -l` para confirmar si `* * * * *` se honra o si Site Tools fuerza mínimo de 15 min. Documentar resultado en consequences negativas de ADR-0001 si es 15-min. (4) **Limpieza one-shot de "REVISION MENSUAL"** en datos legacy: correr el SELECT inventario de ADR-0004 §"Limpieza puntual", revisar lista con cliente, ejecutar `UPDATE asignacion SET tipo=2 WHERE id IN (...)` con backup previo confirmado, documentar en `docs/security.md`. |
| 03 | `03-eloquent-models.md` | Genera Eloquent Models para las 14 tablas legacy con `$table`, `$fillable`, `$casts`, `$timestamps=false` donde aplique, y relaciones (HasMany, BelongsTo) según el modelo de dominio. **Accessors/mutators de charset latin1↔utf8mb4** en campos de texto (`notas`, `direccion`, `nombre_completo`, etc.). |
| 04 | `04-lv-internal-tables.md` | Migrations para tablas internas Laravel con prefijo `lv_`. **Tabla `lv_users` con schema de ADR-0005** (`legacy_kind`, `legacy_id`, `email` no único, `password` nullable, `legacy_password_sha1`). Migrar. |
| 05 | `05-filament-install.md` | `composer require filament/filament`. Crear primer admin user vía tinker (insertando manualmente fila en `lv_users` con `legacy_kind='admin'` apuntando a `u1.id`). Acceder al panel. **Custom theme** que aplica DESIGN.md (cobalto + Instrument Serif). |
| 06 | `06-auth-sha1-bcrypt.md` | Implementa `LegacyHashGuard::attempt()` con la lógica lazy completa de ADR-0003 + ADR-0005: **(a) rate limit `RateLimiter::tooManyAttempts(5, 60s)`** antes de cualquier query (Fortify throttle no aplica fuera de su pipeline). **(b) lookup canónico por `(legacy_kind, legacy_id)`** tras resolver legacy, NUNCA por email en lv_users. **(c) hash_equals** para SHA1, **(d) re-lookup legacy si bcrypt falla**. **(e) role-hint por subdominio/ruta**. Tests obligatorios completos según copilot-instructions.md (incluido `lookup_canonical_by_legacy_kind_legacy_id` y `login_throttles_after_5_failures`). |
| 07 | `07-filament-piv-resource.md` | Resource Filament para `Piv`: tabla con búsqueda, filtros (status, operador, municipio), formulario con tabs. **Eager loading obligatorio** vía `getEloquentQuery()->with(['operador','modulo'])` + test `expectQueryCount`. Validación `municipio` con `Rule::exists('modulo','id')`. |
| 08 | `08-filament-averia-asignacion.md` | Resources para `Averia` y `Asignacion` con relación a `Piv` y filtros por fecha/tipo/status. |
| 09 | `09-filament-cierre.md` | Action en Filament para cerrar asignación. **Field mapping correcto** (corregido tras review externa, ver ADR-0004): tipo=1 form tiene tres campos `Diagnóstico → correctivo.diagnostico`, `Acción → correctivo.accion`, `Foto → correctivo.imagen`. **NUNCA toca `correctivo.notas`** (la columna no existe en el schema legacy) **ni reescribe `averia.notas`** (ya está rellenada por el operador al reportar). tipo=2 abre form `Revision` con checklist por columnas. Imposible escribir en la tabla equivocada. Tests obligatorios completos. |
| 10 | `10-dashboard-reporting.md` | Dashboard Filament con widgets: top paneles con incidencias filtrando por `tipo=1` + filtro de notas; distribución mensual; tiempo medio resolución. **Export CSV/PDF a través de `TecnicoExportTransformer`** que filtra los 6 campos RGPD prohibidos (DNI, NSS, CCC, teléfono, dirección, email). Tests obligatorios `tecnico_export_blacklist`. |
| 11 | `11-tecnico-pwa.md` | Vistas Livewire+Volt mobile-first para técnico. **Dos botones-tarjeta apilados** según ADR-0004 + sección 10.1 de DESIGN.md (stripe izquierdo rojo/teal, tinte de fondo, icono semántico, subtítulo desambiguador). PWA manifest + service worker via **`vite-plugin-pwa` con `registerType: 'prompt'`** (NO autoUpdate — review externa flag: autoUpdate puede invalidar formularios en cola con CSRF mismatch silencioso al deployear cambios de session). Componente Livewire que escucha el evento `pwa:update-available` y renderiza banner persistente "Nueva versión disponible — Recargar". El técnico decide cuándo recargar. **Middleware obligatorio** que detecta `lv_users.legacy_id` apuntando a fila legacy borrada → logout limpio con mensaje "Cuenta desactivada". |
| 12 | `12-operador-pwa.md` | Vistas Livewire+Volt para operador: lista de paneles propios filtrada por `operador_id IN (op_id, _2, _3)`, reportar avería con foto, ver histórico. **Tests obligatorios** de aislamiento entre operadores. |
| 13 | `13-notifications-push.md` | Web Push (`laravel-notification-channels/webpush`) para notificar al técnico nueva asignación. VAPID en `.env`. **Permitir `dispatchSync()`** en la action "Asignar y notificar ahora" del admin (sin esperar al cron). |
| 14 | `14-crons-scheduler.md` | Migrar `cron/asignarMantenimiento.php` y `cron/enviarAsignaciones.php` legacy a Laravel Scheduler. Cron único en SiteGround llama a `php artisan schedule:run` cada minuto (verificado en Bloque 02). **Idempotencia obligatoria**: `AsignarMantenimientoMensual` usa `firstOrCreate` por `(piv_id, fecha del día)` para que correrlo dos veces el mismo día sea no-op. Test `cron_asignar_mantenimiento_idempotent_same_day`. |
| 14b | `14b-images-prune.md` | Comando `php artisan lv:images-prune` que reconcilia FS↔BD: lista archivos en `images/piv/` sin fila en `piv_imagen` (fotos huérfanas) y filas en `piv_imagen` con `path` que no existe en disco (refs rotas). Programado mensual con email de reporte al admin. Resuelve los 375 huérfanos + 2 refs rotas documentados. |
| 14c | `14c-verify-legacy-pointers.md` | Comando `php artisan lv:verify-legacy-pointers` que recorre todas las filas de `lv_users` y verifica que `(legacy_kind, legacy_id)` resuelve a fila viva en su tabla origen. Si encuentra punteros muertos (técnico borrado en app vieja, etc.), emaila al admin con la lista para acción manual. Programado **semanal**. Test obligatorio `lv_verify_legacy_pointers_emails_admin_on_dangling`. |
| 15 | `15-deploy-siteground.md` | Pipeline manual: Mac → GitHub → SSH → `git pull` → `composer install --no-dev --optimize-autoloader` → `php artisan migrate --force` → `optimize`. Sin downtime de la app vieja. |
| 15b | `15b-staging-environment.md` | Subdominio `staging.piv.winfin.es` en SiteGround GoGeek. BD `dbvnxblp2rzlxj_staging` con dump diario de prod restaurado vía cron, **anonimizado** (emails reemplazados por `staging+{id}@winfin.local`, passwords reemplazados por hash conocido de "staging"). `.env.staging` separado. Pipeline: Mac → GitHub → staging → smoke test → prod. **Crear `docs/runbooks/restore.md`** con: comandos exactos SSH para restaurar dump SiteGround diario, comando `mysql --defaults-extra-file` para importar, decryption del 7z mensual de Dropbox (passphrase en gestor), queries de verificación post-restore, RTO objetivo (4h), RPO (24h), **segundo poseedor de la passphrase del 7z** (firma legal de Winfin Systems, no solo el propietario — single point of failure removed). **Primera prueba de restore real ejecutada antes** de marcar el bloque done, no diferida 6 meses. |
| 16 | `16-cutover.md` | Fase 7 cierre: redirects 301 `winfin.es/*.php` → `piv.winfin.es`; mover archivos viejos fuera de `public_html`; actualizar `pagina_web/` con enlaces nuevos; purgar `lv_users.legacy_password_sha1 IS NOT NULL` con notificación previa a usuarios dormidos. (El borrado del dump SQL público se ejecutó en Bloque 02, no aquí.) |
